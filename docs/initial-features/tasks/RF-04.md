## 1. Objetivo (o que este RF entrega)

Garantir que **só pessoas autorizadas** usem o sistema (login obrigatório via Clerk, Google como único método) e que cada empacotamento possa registrar **quem o fez**. Tecnicamente: configura o `proxy.ts` (Next 16) com `clerkMiddleware`, o `<ClerkProvider>` no layout, sign-in custom via hooks, página `/sso-callback` e um helper `requireAuth()` que cada `route.ts` usa para barrar requisições sem sessão (`401 UNAUTHENTICATED`).

## 2. Contexto (por quê)

A loja vende ao vivo no TikTok e os dados de venda/empacotamento são compartilhados entre operadores em aparelhos diferentes. O acesso precisa ser fechado: sem login, nenhuma tela é acessível (RN-7.1). Além disso, o `operatorId` do `Packing` será o `userId` do Clerk — ou seja, esta base de auth é pré-condição para registrar autoria do empacotamento mais adiante (RF-14). Sem papéis: todos os usuários logados têm o mesmo poder (D7).

## 4. Escopo — entregáveis (faça exatamente isto)

- `bun add @clerk/nextjs` — pacote oficial do Clerk para Next.
- **Env:** documentar `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` e `CLERK_SECRET_KEY` (`.env.example`).
- `src/proxy.ts` (**Next 16 — NÃO `middleware.ts`**): `clerkMiddleware` protegendo **todas** as páginas e rotas de API; públicas **apenas** `/(sign-in|sso-callback)`; usar `await auth.protect()` para as protegidas.
- `src/app/layout.tsx` (editar; segue Server Component): envolver a árvore com `<ClerkProvider>`.
- **Sign-in custom** (UI própria, **sem componentes prontos** do Clerk): página de sign-in usando `useSignIn().sso({ strategy: "oauth_google", ... })`, estilizada com nossos primitivos (Tailwind).
- **`/sso-callback`:** página que finaliza o fluxo OAuth do Google.
- **`requireAuth()`** (helper de rota): em cada `route.ts`, além do `proxy.ts`, confirmar `const { isAuthenticated, userId } = await auth()`; sem auth → `401 UNAUTHENTICATED`. `operatorId` do `Packing` = `userId` do Clerk.
- `src/lib/error-messages.ts`: **acrescentar** a entrada PT amigável para `UNAUTHENTICATED` (nunca texto cru do Clerk).
- **Google-only (config de dashboard, não código):** registrar como nota — no painel do Clerk habilitar Google SSO e desabilitar e-mail/senha; cadastro fechado (convites + criação manual).

### Fora do escopo (NÃO faça)

- Não criar o **Modal de Configurações** nem ler perfil (`useUser()`/`signOut()`) — isso é RF-11 (front).
- Não alterar `prisma/schema.prisma` (criado só pelo RF-01).
- Não implementar nenhum caso de uso/rota de domínio (catálogo, pedidos etc.).
- Não usar componentes prontos do Clerk (`<SignIn/>` etc.) — a UI é custom via hooks.

## 5. Fim — Definition of Done

- Sem login, **nenhuma** página/rota de API abre; rota protegida devolve **`401 UNAUTHENTICATED`** sem sessão.
- Login Google funciona via sign-in custom + `/sso-callback`; `proxy.ts` deixa públicas só `/(sign-in|sso-callback)`.
- `requireAuth()` disponível e usado como padrão pelas rotas; `userId` do Clerk acessível para virar `operatorId`.
- Erros de auth viram **toast amigável PT** (entrada `UNAUTHENTICATED` no dicionário).
- `bun run lint` limpo; sem `any`.

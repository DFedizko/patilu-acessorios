## 1. Objetivo (o que este RF entrega)

Fazer a página **Categorias** funcionar de verdade: CRUD de categorias e faixas persistido no banco (não mais em memória), com o **código de barras gerado pelo servidor** e a **impressão da etiqueta** da faixa. Tecnicamente: refatorar `src/app/categories/page.tsx` e os componentes em `src/components/categories/*` para chamar a API via **service → hook (query/mutation)**, removendo o seed do catálogo (`use-catalog-store`), e imprimir a etiqueta via `GET /api/tiers/:id/label` + `window.print()` (+ `@page`).

## 2. Contexto (por quê)

Hoje a tela de Categorias é completa, porém roda 100% com **dados de mentira em memória** (Zustand `use-catalog-store` + `seed.ts`): nada persiste, nada sincroniza entre aparelhos e o código de barras é falso (`789xxxx`). O back-end de catálogo já existe (RF-06/RF-07) e a etiqueta SVG real (RF-08), e o primitivo `Modal` está pronto (RF-11). Este RF liga a tela à API real, dando à loja o catálogo persistido e compartilhado que o empacotamento e os relatórios consomem. É **front**: telas via service/hook, erros via toast; sem testes automatizados.

## 7. Trecho da PRD que fala deste RF

> ### F1 — Categorias e faixas de custo (CRUD)
>
> **Requisitos:**
>
> - Criar categoria (só nome).
> - Renomear e excluir categoria.
> - Criar faixa dentro de uma categoria (nome + custo).
> - Editar faixa (nome e/ou custo).
> - Excluir faixa.
> - Ao criar uma faixa, o **sistema gera automaticamente um código de barras único** para ela (o operador não digita código).
> - A partir da faixa, **imprimir a etiqueta/cartela** do código de barras (fluxo de impressão detalhado em **F6**).

> **Regras de negócio:**
>
> - **RN-1.1** Excluir uma categoria que tem faixas **não bloqueia**: as faixas vão para **"Sem categoria"** (uma categoria implícita/padrão, sempre presente).
> - **RN-1.2** O código de barras de cada faixa é **único** e gerado pelo sistema; não é editável pelo operador.

F6 (imprimir a etiqueta, RN-6.4):

> - **Imprimir** a etiqueta da faixa **conectando-se à impressora**, com dois modos **configuráveis**:
>     - **Impressão direta** numa impressora de etiquetas (térmica) já configurada — idealmente sem diálogo a cada impressão, para gerar várias cartelas rápido.
>     - **Diálogo do navegador** como alternativa, quando não há impressora configurada (serve para qualquer impressora).

> - **RN-6.4 (impressão configurável):** o modo de impressão (direta vs diálogo do navegador) é uma configuração; sem impressora configurada, cai no diálogo do navegador.

Estado atual (gap front × back, §9):

> - **Front-end:** completo e funcional, mas usa **dados de mentira em memória** (Zustand) — catálogo, caixas e Ads são _seed_ (…). Nada persiste; nada sincroniza; não há login.

## 8. Trecho da TECH_SPEC que fala deste RF

Telas afetadas (§13.4):

> - **Sem stores de seed:** trocar Zustand-seed por hooks de query/mutation reais (o catálogo/pedidos/ads passam a vir da API). Zustand fica só para estado de UI genuinamente compartilhado (ex.: rascunho do empacotamento, modal de etiqueta).

Estrutura por camada (§13.1):

> `component → hook (query|mutation) → service → HttpClient → API`. DI funcional via Inversify (`src/di/container.ts` + `symbols.ts`): services (factory functions) recebem o **`HttpClient`** (factory `axiosHttpClient`, §7.4) por injeção; hooks pedem a interface do service. Resolver pelo container dentro da factory. Nenhum service chama `axios`/`fetch` direto — sempre via `HttpClient`.

> **Localização das camadas no front:** a camada de **service fica em `src/service/`** (diretamente sob `src/`, **não** dentro de `src/hooks/`). `src/hooks/query/` e `src/hooks/mutation/` ficam separados e dependem da interface do service.

Tratamento de erros (§13.2):

> - `QueryClient` com `onError` global (queries e mutations) → `resolveMessage(error)` → `toast.error(...)` (sonner). Hooks específicos podem sobrescrever quando precisarem de UX diferente.

Código de barras — render/impressão (§11):

> - **Render da etiqueta (imagem):** `IBarcodeRenderer` → `BwipBarcodeRenderer` usa **bwip-js** `toSVG({ bcid: "code128", text, includetext: true })` no servidor. Endpoint `GET /api/tiers/:id/label` devolve `image/svg+xml`. A página de etiqueta é Server Component; só o botão de imprimir é client.
> - **Impressão:** **MVP** = `window.print()` + CSS `@media print` com `@page { size: 50mm 30mm; margin: 0 }`. **Depois (QZ Tray)** = impressão silenciosa ZPL/ESC-POS via agente desktop (config de impressora moraria no modal de Configurações). Ver risco §17.

Endpoints de catálogo (§8):

> | GET    | `/api/categories`                    | Lista categorias com suas faixas (+ "Sem categoria")            | —                                               | `CategoryWithTiers[]`                     |
> | POST   | `/api/categories`                    | Cria categoria                                                  | `{ name }`                                      | `Category` (201)                          |

## 1. Objetivo (o que este RF entrega)

O CRUD de **faixas de custo** com **código de barras único gerado pelo sistema**: criar (gera o código com retry de unicidade), editar, excluir e resolver um código bipado para a faixa correspondente. Tecnicamente: o domain service `BarcodeCodeGenerator`, 4 casos de uso `*UseCase`, as rotas `/api/tiers*` e `/api/categories/:id/tiers`, schemas Zod e OpenAPI.

## 2. Contexto (por quê)

Cada faixa representa um preço de compra e recebe **um** código de barras único, não editável pelo operador (RN-1.2 / D6). Esse código é a base do recurso de bipe (F6): bipar a cartela de uma faixa equivale a apertar **+1** daquela faixa na tela de empacotamento — e o item entra com a categoria e o custo daquela faixa no momento (RN-6.1/RN-6.2). `FindTierByBarcodeUseCase` é o que o bipe consulta. Editar/excluir uma faixa **nunca** altera empacotamentos já feitos (RN-1.3 — histórico congelado, garantido pelo snapshot no RF-14).

## 4. Escopo — entregáveis (faça exatamente isto)

**Casos de uso** (`src/server/application/use-case/` + contrato em `.../contracts/`, padrão §6.1):

- `CreateTierUseCase` (DTO) — `{ categoryId?, name, costReais }` → Tier (gera barcode, persiste; em violação `@unique`, **regenera** e tenta de novo). Erros: `TIER_COST_MUST_BE_POSITIVE`, `CATEGORY_NOT_FOUND`.

- `FindTierByBarcodeUseCase` — `{ barcode }` → Tier (resolve bipe). Erro: `TIER_NOT_FOUND`.

**Rotas** (`src/app/api/`, borda HTTP: Zod + Clerk + `toHttpResponse`):

## 7. Trecho da PRD que fala deste RF

> ### F1 — Categorias e faixas de custo (CRUD)
>
> **Requisitos:**
>
> - Criar faixa dentro de uma categoria (nome + custo).
> - Editar faixa (nome e/ou custo).
> - Excluir faixa.
> - Ao criar uma faixa, o **sistema gera automaticamente um código de barras único** para ela (o operador não digita código).

> **Regras de negócio:**
>
> - **RN-1.2** O código de barras de cada faixa é **único** e gerado pelo sistema; não é editável pelo operador.
> - **RN-1.3** Excluir ou editar uma faixa **nunca** altera caixas já fechadas (ver RN-3.3 — histórico congelado).
> - **RN-1.4** Custo de faixa deve ser **maior que zero**.

F6 — bipe (semântica):

> - **RN-6.1 (um código por faixa):** cada faixa tem exatamente um código de barras único; bipá-lo soma **+1** daquela faixa (igual ao botão **+**).
> - **RN-6.2 (origem do item bipado):** o item somado é o da **faixa cujo código foi bipado**, com a **categoria e o custo daquela faixa** no momento — entra no empacotamento como qualquer item contado (e congela ao concluir, RN-2.8).
> - **RN-6.3 (código não encontrado):** se o código não corresponder a nenhuma faixa, mostrar mensagem amigável e **não** travar — o operador continua pelo toque (mesma regra de RN-2.9).

> | D6  | Código de barras da faixa é **gerado pelo sistema** e único.                                                                                                                               |
> | D12 | Código de barras é **por faixa** (uma cartela na bancada), não por unidade física; **bipar = +1** daquela faixa.                                                                           |

Código de barras — geração (§11):

> - **Geração do código (dado):** `BarcodeCodeGenerator` (nanoid, alfabeto sem `0/O/1/I/L`, 10 chars, prefixo `T`). `CreateTierUseCase` gera, tenta persistir; em violação de `@unique`, regenera (retry curto) → garante unicidade. Erro final improvável → `DUPLICATE_BARCODE`.
> - **Bipe (scanner keyboard-wedge):** hook client `useBarcodeScanner(onScan)` (`@/hooks/use-barcode-scanner.ts`) escuta `keydown` no `window`, acumula caracteres rápidos e dispara `onScan(code)` no Enter, descartando digitação humana (intervalo > ~50ms) e bursts < 4 chars. Na tela de empacotamento, `onScan` resolve via `GET /api/tiers/by-barcode/:code` e faz `incrementTier` no rascunho (toque continua funcionando — RN-2.9).
> - **Libs:** `bun add bwip-js nanoid`.

export const createTierSchema = z.object({

export type CreateTierDTO = z.input<typeof createTierSchema>; // → contracts/ICreateTierUseCase Input
